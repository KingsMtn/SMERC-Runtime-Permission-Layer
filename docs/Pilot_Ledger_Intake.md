# Pilot Ledger Intake

Pilot Ledger Intake appends real pilot evidence to an existing SMERC Decision Lifecycle Ledger.

The goal is to move from decision-time proxy evidence toward pilot evidence without changing the DLL contract.

## Inputs

The intake path accepts:

- an existing DLL JSON file, or
- the benchmark DLL bundle generated by `reference_engine.benchmark_ledger_builder`

It also accepts a pilot intake JSON file containing appendable evidence:

- `HUMAN_INTERACTION`
- `EXECUTION`
- `OUTCOME`
- `LEARNING_RECOMMENDATION`

## Guardrails

The intake engine rejects:

- evidence for the wrong decision ID
- evidence for the wrong tenant
- invalid source ledgers
- non-appendable event types
- outcome records before execution records
- learning recommendations before outcome records
- duplicate execution, outcome, or learning records through this path
- learning records that try to auto-activate policy changes

## Example

```bash
python -m reference_engine.pilot_ledger_intake \
  reports/runtime_benchmark_dll_bundle.json \
  examples/pilot_ledger_intake_example.json \
  --decision-id "dll:proxy-deploy-001::baseline" \
  --json-output reports/pilot_ledger_intake_result.json \
  --markdown-output reports/Pilot_Ledger_Intake_Result.md \
  --ledger-markdown-output reports/Pilot_Completed_DLL_Example.md \
  --pretty
```

## Evidence Boundary

The included example is synthetic pilot-style evidence. It demonstrates the intake workflow; it is not customer evidence.

In a real design-partner pilot, this path should be fed by reviewer labels, execution reports, rollback evidence, incident data, and post-action outcome review.

This module does not provide legal recordkeeping, immutable storage, SIEM replacement, compliance certification, or production retention controls.
